# card-mizer Productization Plan

기준일: 2026-04-18

## Document Purpose

이 문서는 `card-mizer`를 포트폴리오/데모 백엔드에서 상용 제품 수준의 서비스로 끌어올리기 위한 실행 계획을 정리한다. 현재 저장소의 강점은 추천 도메인 모델과 유스케이스 구조다. 상용화의 핵심 과제는 사용자 단위 데이터 모델, 운영 안정성, 카드 정책 운영 체계, 보안과 규제 대응, 지속 배포 체계를 붙이는 것이다.

## Current Baseline

- `card-core`에 카드 추천, 실적 조회, 사용 내역 기록, 카드 정책 관리 유스케이스가 구현돼 있다.
- `card-api`는 REST API, 정적 데모 UI, 카드사 API 시뮬레이터를 함께 제공한다.
- `card-infra`는 JPA 기반 영속성과 외부 카드사 연동 어댑터를 제공한다.
- 기본 실행 경로는 `h2`이며, PostgreSQL 프로파일과 통합 테스트 경로가 준비돼 있다.
- GitHub Actions CI가 `./gradlew test`와 `:card-api:postgresIntegrationTest`를 수행한다.
- 현재 서비스는 단일 사용자 데모에 가깝고, 상용 서비스에 필요한 인증, 권한, 사용자 데이터 격리, 운영 배포, 모니터링, 감사 추적, 카드 정책 운영 프로세스는 부족하다.

## Product Assumption

- 1차 상용 버전은 한국 시장 기준의 카드 추천/실적 관리 서비스로 본다.
- 초기 출시 형태는 "인증된 사용자가 자신의 카드와 사용 내역을 연결하고, 다음 결제 추천과 실적 알림을 받는 웹 기반 서비스"로 둔다.
- 카드사 직접 제휴나 정산 기능은 1차 범위에서 제외한다.
- 모바일 앱은 1차 범위에서 제외하고, 웹 앱과 API를 먼저 안정화한다.

## Business Goal

- Closed Beta에서 실제 사용자 데이터를 기반으로 추천 정확도와 잔존 가치를 검증한다.
- GA 시점에는 "실적 추적", "다음 결제 추천", "실적 임박 알림", "카드 정책 운영"이 신뢰 가능한 수준으로 동작해야 한다.
- 제품 KPI는 다음 세 가지로 단순화한다.
  - 주간 활성 사용자 대비 추천 API 사용 비율
  - 추천 수용률 또는 추천 후 실제 결제 반영률
  - 실적 임박 알림 후 실적 달성 전환율

## Productization Gaps

### Domain and Product

- 사용자, 카드, 사용 내역, 추천 결과가 아직 멀티테넌트 또는 멀티유저 기준으로 분리돼 있지 않다.
- 카드 정책 데이터가 코드/fixture 중심이라 운영자가 정책을 수정하고 검수하는 체계가 부족하다.
- 추천 이유는 제공되지만, 상용 서비스 수준의 설명 가능성, fallback 정책, 품질 검증 지표는 부족하다.
- 카드사 연동이 시뮬레이터 중심이며, 실제 외부 연동에서 필요한 인증 갱신, 재시도, 장애 격리 전략이 정리돼 있지 않다.

### Platform and Operations

- 기본 실행 경로가 `h2`이므로 운영 기준 데이터베이스, 마이그레이션, 백업, 복구 전략이 아직 미완성이다.
- 인증, 권한, 감사 로그, 개인정보 마스킹, 비밀정보 관리 체계가 없다.
- 메트릭, 로그 상관관계, 에러 알림, SLO, 운영 대시보드가 없다.
- 스테이징/프로덕션 배포 파이프라인, 롤백 절차, 운영 체크리스트가 없다.

### Delivery and Quality

- 추천 품질을 지속 검증하는 회귀 테스트 데이터셋과 운영 데이터 기반 평가 절차가 없다.
- API 버전 정책과 호환성 정책이 정의되지 않았다.
- 장애 대응 문서, 온콜 기준, 고객 지원 흐름이 없다.

## Release Strategy

### Phase 0. Product Definition and Risk Reduction

목표: 제품 방향과 운영 범위를 먼저 고정한다.

- 대상 사용자와 핵심 사용 시나리오를 3개로 축소한다.
- MVP 범위를 "카드 등록", "거래 동기화", "실적 조회", "추천", "알림"으로 확정한다.
- 추천 엔진의 핵심 판단 원칙과 우선순위를 ADR로 문서화한다.
- 카드 정책 데이터의 소스 오브 트루스를 코드 fixture에서 운영 가능한 저장 방식으로 옮길 계획을 확정한다.
- 개인정보, 카드 데이터, 외부 연동 자격증명에 대한 보안 등급과 저장 정책을 정의한다.

완료 기준:

- 제품 요구사항 문서 1개
- 핵심 ADR 3~5개
- MVP API 목록과 비기능 요구사항 확정

### Phase 1. Single-Tenant Production Beta

목표: 실제 사용자 한 팀 또는 소규모 베타 고객이 쓸 수 있는 수준으로 만든다.

- PostgreSQL을 기본 운영 저장소로 승격하고 `h2`는 로컬 데모 전용 프로파일로 내린다.
- 사용자 계정, 세션, 카드 소유권, 사용 내역 소유권 모델을 추가한다.
- 최소한의 인증 방식(OAuth2 또는 이메일 매직링크)과 역할 구분(User/Admin)을 도입한다.
- 카드 정책/카드 메타데이터 관리용 관리자 API를 추가한다.
- 거래 동기화 실패, 중복 반영, 부분 실패에 대비한 idempotency와 재처리 전략을 넣는다.
- 추천 요청/응답, 정책 변경, 동기화 작업에 대한 감사 로그를 추가한다.

완료 기준:

- 인증된 사용자 기반 end-to-end 흐름 동작
- PostgreSQL 기준 마이그레이션과 롤백 절차 검증
- 관리자 권한으로 카드 정책 수정 가능
- 거래 동기화와 추천 API의 핵심 경로 통합 테스트 확보

### Phase 2. Operational Readiness

목표: 제품이 아니라 서비스로 운영할 최소 조건을 맞춘다.

- Prometheus, Grafana, structured logging, request correlation ID를 도입한다.
- 핵심 메트릭을 정의한다.
  - 추천 요청 수와 실패율
  - 추천 계산 시간
  - 거래 동기화 성공률
  - 알림 발송 성공률
  - 사용자별 최신 실적 계산 지연
- 에러 추적과 운영 알림 채널을 붙인다.
- 스테이징/프로덕션 환경 분리, 비밀정보 주입, DB 백업/복구, 헬스체크를 정리한다.
- 배포 파이프라인에 smoke test와 rollback 기준을 추가한다.

완료 기준:

- 운영 대시보드 1세트
- 배포 체크리스트와 장애 대응 문서
- 스테이징에서 실사용 시나리오 리허설 완료

### Phase 3. Recommendation Quality and Trust

목표: "돌아간다"가 아니라 "믿고 쓸 수 있다" 수준으로 끌어올린다.

- 추천 결과에 대해 설명 가능한 이유, 대안 카드, 추천 불확실성 또는 fallback 사유를 표준화한다.
- 카드 정책 변경이 추천 품질에 미치는 영향을 검증하는 회귀 데이터셋을 만든다.
- 카드사별 거래 정규화 규칙을 운영 데이터 기반으로 보강한다.
- 추천 수용률과 실적 달성률을 측정하는 분석 이벤트 체계를 넣는다.
- 추천 실패 시 단순 혜택 우선, 실적 달성 우선 같은 안전한 대체 전략을 정의한다.

완료 기준:

- 회귀 테스트 데이터셋과 품질 리포트 자동 생성
- 추천 이유 포맷 고정
- 운영에서 품질 이상 징후를 볼 수 있는 지표 확보

### Phase 4. GA and Monetization Readiness

목표: 공개 서비스 또는 유료 베타로 전환할 수 있는 상태를 만든다.

- 이용약관, 개인정보 처리 흐름, 데이터 삭제 요청 처리 절차를 마련한다.
- 고객 지원과 문의 대응 흐름을 정리한다.
- 서비스 요금제 또는 무료/유료 기능 구분을 정한다.
- 사용자 온보딩, 카드 등록 유도, 첫 추천 경험을 개선한다.
- 공개 API 정책을 유지할지, 웹 제품 중심으로 갈지 결정한다.

완료 기준:

- 베타 운영 결과를 반영한 정식 출시 체크리스트
- 고객 지원과 데이터 삭제 절차 문서화
- 제품 KPI와 수익화 가설 검토 완료

## Cross-Functional Workstreams

### 1. Product and UX

- 정적 데모 UI를 상용 웹 앱 수준으로 대체한다.
- 첫 사용자 온보딩, 카드 등록, 추천 결과 확인, 알림 설정 흐름을 설계한다.
- 실패/빈 결과/동기화 지연 상태를 사용자에게 설명 가능한 화면으로 바꾼다.

### 2. Data and Policy Operations

- 카드 상품 메타데이터와 실적 정책을 별도 관리 테이블 또는 관리 서비스로 분리한다.
- 정책 변경 이력, 검수, 롤백 체계를 둔다.
- 카드사 정책 업데이트 주기와 검수 담당 절차를 만든다.

### 3. Security and Compliance

- 민감정보 최소 수집 원칙을 정한다.
- 사용자 식별자, 토큰, 외부 연동 자격증명 암호화 정책을 적용한다.
- 감사 로그, 접근 제어, 관리 기능 권한 분리를 넣는다.

### 4. Reliability

- 배치/동기화 작업에 재시도, idempotency, dead-letter 또는 보류 큐 전략을 적용한다.
- 추천 엔진 실패 시 API 전체 장애로 번지지 않도록 fallback 경로를 둔다.
- 운영 장애 시 읽기 전용 모드 전환 기준을 마련한다.

## Prioritized Backlog

### P0

- 운영 저장소를 PostgreSQL로 통일
- 인증/인가 추가
- 사용자 단위 데이터 모델 도입
- 카드 정책 운영 구조 확정
- 추천 스코어링 ADR 작성

### P1

- 거래 동기화 안정화
- 실적 임박 알림 도입
- 관리자 정책 UI 또는 관리자 API 구축
- 운영 메트릭/로그/알림 도입
- 배포 파이프라인과 스테이징 환경 구축

### P2

- 추천 품질 회귀 데이터셋 자동화
- 분석 이벤트 수집
- 고객 지원 운영 문서
- 수익화 실험

## Non-Goals for First Commercial Release

- 카드 발급 또는 결제 실행 대행
- 복수 국가 지원
- 복잡한 금융상품 비교 전반
- 네이티브 모바일 앱 동시 출시
- 대규모 파트너 공개 API 생태계

## Exit Criteria

다음 조건을 만족하면 1차 상용 출시 후보로 본다.

- 인증된 사용자가 자신의 카드/거래 데이터를 기준으로 추천을 받을 수 있다.
- PostgreSQL, 백업, 모니터링, 알림, 배포 파이프라인이 운영 가능 수준이다.
- 카드 정책 변경이 코드 수정 없이 운영 절차로 반영 가능하다.
- 추천 품질 회귀 테스트가 릴리스 게이트로 동작한다.
- 장애 대응, 데이터 삭제, 고객 문의 처리 문서가 준비돼 있다.

## Immediate Next 6 Weeks

1. 사용자/권한 모델과 PostgreSQL 기준 스키마를 확정한다.
2. 추천 스코어링, 정책 저장 방식, 외부 연동 경계를 ADR로 고정한다.
3. 카드 정책 운영용 관리자 API 초안을 만든다.
4. 실적 임박 알림의 도메인 이벤트와 스케줄링 모델을 구현 계획 수준까지 구체화한다.
5. 운영 메트릭 초안을 정의하고 Prometheus 도입 범위를 결정한다.
