# 2026-03-31 작업 복기 노트

이 문서는 2026-03-31에 `card-mizer`에서 실제로 구현하고 검증한 내용을 나중에 혼자 다시 따라가며 공부할 수 있게 정리한 학습용 메모다.

## 한 줄 요약

오늘 작업으로 `card-mizer`는 추천 전용 데모를 넘어, 카드 등록, 우선순위 조정, 카드 정책 조회/전체 교체/부분 수정, JPA/Flyway 기반 persistence, PostgreSQL 실서버 부팅 검증, 정적 UI 관리 화면까지 갖춘 실행 가능한 데모가 됐다.

## 오늘 끝난 범위

1. 카드 등록 API 추가
2. 카드 우선순위 변경 API 추가
3. 카드 정책 조회/전체 교체 API 추가
4. 입력 검증과 구조화된 에러 응답 추가
5. JPA/H2 기반 persistence로 전환
6. Flyway migration과 PostgreSQL 프로파일 추가
7. PostgreSQL 16 실서버 부팅 검증
8. 카드 정책 `PATCH` API 추가
9. 정적 UI에서 카드/정책 관리 흐름 연동

## 작업 순서와 핵심 포인트

### 1. 카드 등록과 우선순위 조정

- 엔드포인트:
  - `POST /api/cards`
  - `PATCH /api/cards/priorities`
- 핵심 이유:
  - 추천만 되는 데모에서 한 단계 나아가, 카드 구성 자체를 API로 관리할 수 있어야 했기 때문이다.
- 핵심 규칙:
  - 새 카드를 등록할 때는 기본 정책도 함께 만들어야 한다.
  - 그렇지 않으면 추천/실적 조회 로직이 정책 없는 카드를 만나 깨질 수 있다.
- 먼저 볼 파일:
  - `card-api/src/main/java/com/jean202/cardmizer/api/cards/CardManagementController.java`
  - `card-core/src/main/java/com/jean202/cardmizer/core/application/RegisterCardService.java`
  - `card-core/src/main/java/com/jean202/cardmizer/core/application/UpdatePriorityService.java`

### 2. 카드 정책 조회와 전체 교체

- 엔드포인트:
  - `GET /api/cards/{cardId}/performance-policy`
  - `PUT /api/cards/{cardId}/performance-policy`
- 핵심 이유:
  - 카드를 추가만 하고 혜택 규칙을 바꾸지 못하면 실제 추천 품질을 올릴 수 없기 때문이다.
- 설계 포인트:
  - 정책은 `tiers + benefitRules` 전체를 하나의 정책 aggregate처럼 다룬다.
  - `PUT`은 부분 수정이 아니라 전체 교체다.
- 먼저 볼 파일:
  - `card-api/src/main/java/com/jean202/cardmizer/api/cards/CardPerformancePolicyController.java`
  - `card-core/src/main/java/com/jean202/cardmizer/core/application/GetCardPerformancePolicyService.java`
  - `card-core/src/main/java/com/jean202/cardmizer/core/application/ReplaceCardPerformancePolicyService.java`

### 3. 입력 검증과 에러 응답 형식

- 바뀐 점:
  - 요청 DTO에 `@Valid`와 필드 제약을 붙였다.
  - 에러 응답을 `code`, `message`, `path`, `fieldErrors[]` 구조로 통일했다.
- 핵심 이유:
  - 잘못된 입력이 500으로 떨어지지 않고, 프런트/UI에서도 어떤 필드가 잘못됐는지 바로 보여줄 수 있어야 했기 때문이다.
- 먼저 볼 파일:
  - `card-api/src/main/java/com/jean202/cardmizer/api/config/ApiExceptionHandler.java`
  - `card-api/build.gradle.kts`
  - 각 controller의 request record

### 4. JPA/H2 기반 persistence 전환

- 이전 기준선:
  - 인메모리 어댑터 중심
- 오늘 바뀐 기준선:
  - 기본 실행은 JPA + H2
  - 시드 데이터는 최초 부팅 시 DB가 비어 있을 때 적재
- 핵심 이유:
  - 앱 재시작마다 상태가 사라지는 데모를 넘어서, HTTP -> use case -> persistence 흐름을 실제 저장소 경로로 검증하기 위해서다.
- 설계 포인트:
  - 도메인 규칙은 그대로 `card-core`에 남긴다.
  - JPA entity/repository/adapter는 `card-infra`로 밀어낸다.
  - API 조립은 `card-api`의 `ApplicationConfiguration`이 담당한다.
- 먼저 볼 파일:
  - `card-api/src/main/java/com/jean202/cardmizer/api/config/ApplicationConfiguration.java`
  - `card-infra/src/main/java/com/jean202/cardmizer/infra/persistence/jpa/JpaPersistenceConfiguration.java`
  - `card-infra/src/main/java/com/jean202/cardmizer/infra/persistence/jpa/JpaCardCatalogAdapter.java`
  - `card-infra/src/main/java/com/jean202/cardmizer/infra/persistence/jpa/JpaCardPerformancePolicyAdapter.java`
  - `card-infra/src/main/java/com/jean202/cardmizer/infra/persistence/jpa/JpaSpendingRecordAdapter.java`

### 5. Flyway와 PostgreSQL 프로파일

- 바뀐 점:
  - 스키마 생성 책임을 Hibernate 자동 생성이 아니라 Flyway migration으로 옮겼다.
  - `h2`와 `postgres` 프로파일을 분리했다.
  - 로컬 PostgreSQL은 `docker compose up -d`로 띄우는 경로를 만들었다.
- 핵심 이유:
  - H2에서만 우연히 되는 동작이 아니라, 실제 PostgreSQL에서도 스키마와 JPA가 맞물리는지 확인하기 위해서다.
- 실제 부딪힌 문제:
  - 처음에는 다른 프로젝트의 PostgreSQL이 `5432`를 점유하고 있었다.
  - `compose.yaml` 포트를 `55432`로 바꿔 충돌을 피했다.
  - Flyway가 PostgreSQL 16.13을 인식하지 못해 `flyway-database-postgresql` 모듈을 추가했다.
- 먼저 볼 파일:
  - `compose.yaml`
  - `card-api/src/main/resources/application.properties`
  - `card-api/src/main/resources/application-h2.properties`
  - `card-api/src/main/resources/application-postgres.properties`
  - `card-api/src/main/resources/db/migration/V1__create_core_tables.sql`
  - `card-api/src/main/resources/db/migration/V2__drop_cards_priority_unique_index.sql`
  - `card-api/build.gradle.kts`

### 6. PostgreSQL 실서버 검증

- 검증 기준:
  - PostgreSQL 컨테이너에 실제로 연결되는가
  - Flyway migration이 실제 DB에 적용되는가
  - JPA 초기화가 성공하는가
  - HTTP 서버가 응답하는가
- 실제 확인한 것:
  - `postgres` 프로파일로 부팅 성공
  - Flyway `v1`, `v2` 적용 성공
  - `http://localhost:18080/` 응답 `200` 확인
- 학습 포인트:
  - “설정 파일을 만들었다”와 “실서버 검증이 끝났다”는 전혀 다른 단계다.
  - 부팅 로그를 끝까지 확인해야 인증 문제, 포트 충돌, Flyway 버전 문제를 실제로 잡을 수 있다.

### 7. 카드 정책 `PATCH`

- 엔드포인트:
  - `PATCH /api/cards/{cardId}/performance-policy`
- 동작 규칙:
  - `tiers`만 보내면 기존 `benefitRules`는 유지
  - `benefitRules`만 보내면 기존 `tiers`는 유지
  - 둘 다 없으면 `400`
- 설계 포인트:
  - 별도 patch용 domain use case를 만들기보다, 현재 정책을 읽어서 API 레이어에서 병합한 뒤 기존 `replace` use case를 재사용했다.
  - 이렇게 하면 저장 규칙은 여전히 하나의 경로로 흘러간다.
- 먼저 볼 파일:
  - `card-api/src/main/java/com/jean202/cardmizer/api/cards/CardPerformancePolicyController.java`
  - `card-api/src/test/java/com/jean202/cardmizer/api/cards/CardPerformancePolicyControllerTest.java`
  - `card-api/src/test/java/com/jean202/cardmizer/api/JpaPersistenceFlowIntegrationTest.java`

### 8. 정적 UI 연동

- 오늘 UI에 붙인 기능:
  - 카드 등록
  - 카드 우선순위 위/아래 이동 후 저장
  - 정책 조회
  - 정책 전체 교체
  - `tiers`만 PATCH
  - `benefitRules`만 PATCH
- 핵심 이유:
  - API가 있어도 화면에서 바로 시연되지 않으면 포트폴리오 가치가 떨어진다.
- 설계 포인트:
  - 별도 프런트 프레임워크 없이, 기존 static HTML/CSS/JS 데모에 관리 패널을 추가했다.
  - 카드 목록 조회 API는 아직 따로 없어서, 현재 선택 월의 `performance-overview` 응답을 관리 기준선으로 재사용했다.
- 먼저 볼 파일:
  - `card-api/src/main/resources/static/index.html`
  - `card-api/src/main/resources/static/app.js`
  - `card-api/src/main/resources/static/app.css`

## 테스트와 검증

오늘 실제로 확인한 주요 명령은 다음과 같다.

```bash
./gradlew test
./gradlew :card-api:bootRun
docker compose up -d
./gradlew :card-api:bootRun --args='--spring.profiles.active=postgres'
./gradlew :card-api:bootRun --args='--server.port=18080'
```

추가로 확인한 것:

- `node --check card-api/src/main/resources/static/app.js`
- PostgreSQL 프로파일로 부팅 후 `http://localhost:18080/` 응답 확인

## 직접 공부할 때 체크할 질문

1. 왜 새 카드 등록 시 기본 정책을 같이 만들어야 하는가
2. 왜 우선순위 변경은 전체 카드 집합을 한 번에 받아야 하는가
3. 왜 `PATCH`를 새 persistence 로직으로 만들지 않고 기존 `replace` 경로를 재사용했는가
4. 왜 도메인 로직은 `card-core`에 남기고, JPA는 `card-infra`로 밀어냈는가
5. 왜 Flyway를 붙인 뒤 PostgreSQL 실서버 검증이 꼭 필요했는가
6. 왜 정적 UI에서도 API 에러 메시지를 그대로 읽어 보여줄 수 있어야 하는가

## 지금 기준 남은 일

- 카드 목록 전용 조회 API가 아직 없다.
- 정책 편집 UI는 JSON 에디터 기반이라 UX를 더 다듬을 여지가 있다.
- 브라우저 기준 E2E 테스트는 아직 없다.
- 운영 환경 기준 PostgreSQL 설정, 마이그레이션 전략, seed 정책은 더 정리할 수 있다.
- 일부 문서는 예전 인메모리 기준선을 설명하고 있어 현재 구현 상태에 맞춘 추가 정리가 필요하다.

## 마무리

오늘 작업의 핵심은 기능 몇 개를 추가한 것보다, “추천 데모”를 “관리 가능한 실행형 데모”로 끌어올린 데 있다. 복기할 때는 endpoint 개수보다 다음 흐름이 머리에 남는지 확인하는 것이 더 중요하다.

`요청 -> controller -> use case -> port -> JPA adapter -> DB -> 다시 UI까지 응답`
